# 产品需求文档：CodePal V1.3.0（版本管理与自动升级）

> 文档版本：v1.3.0-r1  
> 创建日期：2026-03-25  
> 对应版本：`v1.3.0`  
> 对应模块：`src/App.jsx`、`src/components/WorkbenchLayout.jsx`、`electron/main.js`、`electron/preload.js`、GitHub Release 工作流

---

## 1. 背景与目标

当前 CodePal 已经具备以下发布基础：

- 使用 GitHub Release 作为统一发布渠道
- 已能自动生成 macOS 与 Windows 安装包
- Release 中已产出 `latest.yml` / `latest-mac.yml`
- 客户端左下角已预留“新版可用”入口样式

但当前升级体验仍停留在“手动下载安装包”阶段：

- 客户端不会主动检查新版本
- 左下角“新版可用”仍是占位 UI
- 用户点击后没有下载、安装、重启升级逻辑
- 用户必须手动打开 GitHub Release 页面下载覆盖安装

V1.3.0 的目标是建立 CodePal 第一版统一版本体系与客户端升级能力，让应用从“纯手动更新”升级为“应用内感知版本并进入升级流程”。

## 1.1 本轮决策摘要

本轮需求对齐后，V1.3.0 明确采用以下产品决策：

- 版本命名统一切换到 `1.x.x`
- 当前稳定版本为 `v1.2.0`
- 当前开发目标版本为 `v1.3.0`
- Windows 与 macOS 必须在同一版本内一起实现版本检查与升级入口
- Windows 目标为完整自动升级闭环
- macOS 目标为同版接入完整升级代码路径，并接受未签名阶段的系统限制
- 若 V1.3.0 上线后仍有问题，通过 `v1.3.1` 等补丁版本继续修复

---

## 2. 版本策略

### 2.1 统一版本口径

从 V1.3.0 开始，CodePal 对外统一使用语义化版本号：

- 当前稳定版：`v1.2.0`
- 本次目标版本：`v1.3.0`
- 后续版本：`v1.3.1`、`v1.4.0` 等

不再以 `V0.17` 这一类阶段号作为对外主版本表达。

### 2.2 版本一致性要求

以下位置必须保持同一版本号：

- `package.json`
- App 内左下角版本显示
- Git tag
- GitHub Release 标题与产物版本号

---

## 3. 核心目标

### 目标 1：建立统一的版本管理体验

- App 能识别当前版本
- App 能检查 GitHub Release 上是否存在新版本
- App 能在 UI 中明确展示“已是最新”或“新版可用”

### 目标 2：建立客户端升级入口

- 用户可从 App 内点击升级入口
- 系统可启动下载与安装流程
- 用户可感知当前处于“检查中 / 下载中 / 可安装 / 失败”等状态

### 目标 3：Windows 与 macOS 同版支持

- Windows 与 macOS 必须在 V1.3.0 同时接入版本检查与升级入口
- 两个平台必须共用 GitHub Release 作为更新源
- 两个平台必须在同一个版本节奏内交付

说明：

“同版支持”指的是两端都接入完整的产品能力与代码路径。  
但 macOS 的最终自动安装体验受签名与公证条件影响，需在 PRD 中明确为平台约束，而不是实现缺失。

---

## 4. 非目标

以下内容不属于 V1.3.0 范围：

- Linux 平台自动升级
- 灰度发布
- 强制更新
- 多更新渠道切换
- 自定义更新服务器
- 应用内完整 Release Notes 历史列表
- “跳过此版本”“稍后提醒”一类二期交互
- 增量更新策略优化

---

## 5. 核心用户问题

当前用户在升级时存在以下问题：

- 不知道当前自己使用的是否是最新版本
- 不知道新版本是否已经发布
- 必须手动打开 GitHub Release 页面
- 必须手动下载并覆盖安装，流程割裂
- Windows 与 macOS 的升级认知不统一

---

## 6. 用户故事

### US-01 普通用户

作为普通用户，我希望 App 能告诉我是否有新版本，这样我不需要自己去 GitHub 查看。

### US-02 普通用户

作为普通用户，我希望点击“新版可用”后就能进入升级流程，而不是再去手动搜索下载包。

### US-03 测试用户

作为测试用户，我希望 Windows 和 macOS 都能通过同一个产品入口感知更新，而不是每个平台都用完全不同的方法。

### US-04 产品维护者

作为维护者，我希望只要发布 GitHub Release，客户端就能基于同一发布源完成版本检查，不需要额外维护更新服务。

---

## 7. 方案概述

V1.3.0 采用以下总体方案：

- 发布源：GitHub Releases
- 客户端更新能力：Electron 主进程接入更新器
- 前端交互：复用现有左下角“新版可用” pill
- 平台策略：
  - Windows：基于 `NSIS + latest.yml` 走完整自动升级链路
  - macOS：接入同一套版本检查与更新逻辑，并补齐更新器所需产物；未签名环境下需接受最终体验受平台限制

---

## 8. 范围定义

### 8.1 版本检查

系统需要具备统一的版本检查能力。

要求：

- App 启动后静默检查一次更新
- 支持用户手动再次触发检查
- 前端可获取当前版本、远端版本、更新状态
- 能区分以下状态：
  - 空闲
  - 检查中
  - 已是最新
  - 发现新版本
  - 下载中
  - 下载完成
  - 安装中
  - 更新失败

### 8.2 升级入口 UI

复用当前侧边栏底部 `新版可用` pill。

要求：

- 无更新时不误报
- 有更新时显示明确入口
- 点击后进入升级流程
- 下载中展示进度或明确提示
- 下载完成后展示“安装并重启”或等价操作
- 失败时给出用户可理解提示

### 8.3 主进程更新能力

主进程负责真实的更新流程。

要求：

- 检查 GitHub Release 更新
- 下载更新包
- 提供下载进度事件
- 下载完成后支持安装更新
- 将错误和状态变化回传给前端

### 8.4 前端与主进程通信

需要新增稳定的更新相关 IPC。

要求：

- 前端可发起检查更新
- 前端可发起下载更新
- 前端可发起安装更新
- 前端可订阅状态、进度、错误变化

### 8.5 发布链路

Release 产物需要满足更新器消费要求。

要求：

- Windows 保持 `NSIS + latest.yml`
- macOS 补齐自动更新所需产物
- 发布工作流继续保持 tag 驱动
- 产物命名尽量唯一清晰，避免重复资产造成识别歧义

---

## 9. 平台交付定义

### 9.1 Windows 交付定义

V1.3.0 对 Windows 的要求是完整闭环：

- 能检查更新
- 能发现新版本
- 能下载更新
- 能安装更新
- 安装后版本号正确变更

Windows 是本版本必须完整打通的平台。

### 9.2 macOS 交付定义

V1.3.0 对 macOS 的要求也是接入完整产品能力：

- 能检查更新
- 能发现新版本
- 能触发下载与安装代码路径
- Release 产物满足更新器消费要求

但需要明确以下平台约束：

- macOS 的自动更新体验依赖签名与公证条件
- 在未签名环境下，代码可以接入，但最终系统安装体验可能受 Gatekeeper 或平台机制影响
- 因此，若 macOS 在未签名阶段无法达到 Windows 同等级的“无感自动安装”，这属于平台前置条件，不视为 V1.3.0 功能缺失

V1.3.0 对 mac 的验收重点是：

- 版本检查能力真实可用
- 更新状态 UI 真实可用
- 升级路径真实接通
- 为未来签名环境保留可直接生效的实现

## 10. 功能需求清单

### FR-1 当前版本获取

- 系统必须能获取当前应用版本号
- 前端必须能读取并展示当前版本

### FR-2 远端版本获取

- 系统必须能从 GitHub Release 获取最新版本信息
- 系统必须能判断当前版本是否落后于远端版本

### FR-3 自动检查更新

- 应用启动后必须自动检查一次更新
- 后续必须支持手动再次检查

### FR-4 更新提示

- 检测到新版本时必须显示更新入口
- 无新版本时不得误报

### FR-5 下载更新

- 用户点击更新入口后，系统必须支持下载更新包
- 下载过程中必须能向前端返回进度或阶段状态

### FR-6 安装更新

- 下载完成后，系统必须允许用户执行安装更新
- 安装完成后应用应进入新版

### FR-7 错误处理

- 检查失败时必须给出可理解提示
- 下载失败时必须给出可理解提示
- 安装失败时必须给出可理解提示
- 错误提示不得仅停留在控制台

### FR-8 多平台发布兼容

- 同一发布版本必须同时支持 Windows 与 macOS
- Release 产物结构必须满足客户端更新器消费要求

---

## 11. 交互要求

### 11.1 入口层

- 沿用左下角 `新版可用` pill
- 初始状态不打扰用户
- 发现新版本时突出但不过度抢焦点

### 11.2 点击后的状态建议

建议至少覆盖以下交互：

- 检查到更新：显示“立即更新”
- 下载中：显示“下载中”与进度
- 下载完成：显示“立即安装并重启”
- 更新失败：显示失败原因并允许重试

### 11.3 错误提示

- 使用现有 Toast 或等价状态提示
- 避免仅在开发者控制台输出，前端无感知

---

## 12. 技术实现建议

### 12.1 依赖与更新器

- 引入 `electron-updater`
- 使用 GitHub Release 作为 publish provider

### 12.2 模块边界

- 主进程服务层：负责检查、下载、安装与事件分发
- preload 层：最小暴露更新相关接口
- 前端状态层：管理更新状态归一化
- Layout 层：负责入口展示与交互触发

### 12.3 发布配置

- Windows 保持 `nsis`
- macOS 需补齐更新器所需 target
- GitHub Actions Release 工作流继续负责生成并上传更新元数据

---

## 13. 风险与约束

### 风险 1：macOS 签名限制

未签名 / 未公证环境下，macOS 自动升级的最终安装体验可能无法达到正式商用品质。

### 风险 2：Release 资产命名歧义

如果同一平台上传多份命名相近但内容相同的安装包，可能增加更新器识别与排查成本。

### 风险 3：状态设计不完整

若未定义完整状态流转，用户会看到按钮但不知道当前在检查、下载还是失败。

### 风险 4：版本号口径不统一

若 `package.json`、Git tag、Release 标题、UI 展示版本不一致，会造成“明明升级了但看起来像没升级”的认知混乱。

---

## 14. 验收标准（DoD）

V1.3.0 发布前，至少满足以下标准：

1. GitHub Release 可稳定生成更新所需产物。  
2. 客户端启动后可自动检查新版本。  
3. 前端可展示真实更新状态，而不是占位 UI。  
4. Windows 客户端可完成一轮真实升级。  
5. macOS 客户端可完成真实版本检查，并具备更新入口与升级路径。  
6. 错误场景下用户可看到明确提示。  
7. App 内版本号、Git tag、Release 版本号保持一致。

## 15. 建议拆分

### 开发任务 A：发布配置完善

- 清理重复 Release 资产问题
- 补足 macOS 更新所需构建目标
- 固化 GitHub Release 产物结构

### 开发任务 B：主进程更新服务

- 接入更新器
- 封装检查、下载、安装与事件

### 开发任务 C：preload 与 IPC

- 新增更新相关接口
- 新增状态 / 进度事件监听能力

### 开发任务 D：前端版本管理 UI

- 接通现有 `新版可用` 按钮
- 增加检查中 / 下载中 / 下载完成 / 失败状态

### 开发任务 E：验证与发布

- Windows 实机升级验证
- macOS 版本检查与升级路径验证
- 发布说明与测试说明补充

---

## 16. 版本结论

V1.3.0 的核心不是“增加一个更新按钮”，而是建立 CodePal 第一版统一版本体系与客户端升级能力。

本版本采用统一版本号 `v1.3.0`，后续若在该能力上修复问题，再通过 `v1.3.1` 等补丁版本迭代。
