# 产品需求文档：CodePal V1.4.2 Debug（Usage 首屏卡顿修复）

> 文档版本：v1.4.2-r1  
> 创建日期：2026-04-15  
> 对应版本：`v1.4.2`  
> 版本类型：`debug / 稳定性修复版`  
> 对应模块：`src/App.jsx`、`src/pages/UsageMonitorPage.jsx`、`src/pages/usage/useUsageData.js`、`src/pages/usage/useUsageHeavyPeriods.js`、`electron/main.js`、`electron/preload.js`、`electron/aggregateUsagePeriodHandler.js`

---

## 1. 背景与问题定义

当前用户反馈的核心问题不是“安装失败”，而是：

- 新用户或首次启动用户进入 App 后，首屏容易出现明显卡顿
- 用量页中的 `累计至今`、`近7天`、`近30天`、部分自定义区间会出现长时间加载
- 用户主观感受是“窗口一直转、点不动、像应用没起来”

本轮排查后确认，问题本质是：

- **首屏默认进入 Usage**
- **Usage 首次挂载时就并发触发多轮重扫描**
- **重扫描结果以原始日志大对象回传到渲染进程，再在前台线程做解析与聚合**

因此，UI 卡死的根因不是 Electron 本身打不开，而是**渲染线程在首屏被重计算与大体积数据传输打满**。

---

## 2. 根因验证结论

本轮已经完成代码与数据层面的验证，结论如下：

### 2.1 入口层根因

- `src/App.jsx` 默认模块原先为 `usage`
- 用户第一次打开 App 时，不会先进入轻页面，而是直接挂载用量页

### 2.2 Usage 首屏根因

- `src/pages/usage/useUsageData.js` 原先在首次挂载时并发预热：
  - `today`
  - `week`
  - `month`
- 这三轮统计会同时扫描本地日志并做聚合

### 2.3 线程模型根因

- `today/week/month/allTime` 的重计算原先主要走前端 `aggregateUsage(...)`
- `aggregateUsage(...)` 会通过 IPC 读取大量原始 `.jsonl` 行数据
- 渲染层收到原始日志后再做 JSON 解析、去重、模型聚合、项目聚合
- 当日志量足够大时，渲染线程被长时间占用，导致界面失去响应

### 2.4 已验证的卡顿量级

实际验证中已经观察到：

- 单周期约 `100MB` 级原始日志时开始出现可感知卡顿
- 单周期约 `150MB` 级时出现明显卡顿
- 单周期约 `300MB` 级时卡顿严重
- 更高量级时可能进入接近“假死”的状态

在 Electron 端的极端数据复现中，已观察到：

- 最大主线程停顿超过 `1s`
- 更重场景超过 `2s`
- 极端场景可超过 `5s`，并导致页面交互超时

这足以支持“首屏卡顿 / 看起来没启动”这一用户反馈。

---

## 3. 本版目标

V1.4.2 Debug 目标是把 Usage 的卡死问题从根上拆开，确保：

### 目标 1：首屏不再默认进入重页面

- App 首次打开时不再自动进入 Usage
- 首屏应优先落在轻页面，保证应用可交互

### 目标 2：Usage 首次进入只做轻量加载

- 首次进入 Usage 时只加载 `today`
- 不再在首屏并发预热 `week/month`

### 目标 3：重周期改为后台汇总，不阻塞主界面

- `近7天`
- `近30天`
- `累计至今`
- `自定义区间`

以上周期在需要计算时进入后台汇总态，不得让整个页面卡死。

### 目标 4：用户必须看到“系统正在算什么”

- 给出明确的后台汇总提示
- 给出真实进度反馈
- 给出已完成天数 / 总天数
- 允许用户离开 Usage 页继续使用其他模块

### 目标 5：避免重任务并发堆叠

- 同一时刻只允许一个重周期任务运行
- 若某重周期还没有结果，不允许再同时启动另一个无缓存重任务

---

## 4. 非目标

以下内容不属于 V1.4.2 范围：

- 费用估算模型重做
- 账单精度对账
- 历史 Usage 图表体系重构
- 跨设备同步 Usage 缓存
- 所有周期都提供真实后端逐步进度条

说明：

V1.4.2 的重点是**稳定性、首屏体验、线程安全与交互可解释性**，不是做 Usage 功能扩展。

## 5. 用户交互方案

### 5.1 启动时默认页面

- 默认首页从 `Usage` 改为 `启动模式`
- 用户首次打开 App 时，必须先进入轻页面

### 5.2 首次进入 Usage

- 页面初次挂载只拉 `today`
- 不再自动启动 `week/month/allTime/custom` 的后台计算

### 5.3 点击重周期时的行为

#### 情况 A：已有缓存且缓存新鲜

- 立即展示结果
- 不显示阻塞态

#### 情况 B：已有缓存但缓存过期

- 先展示旧结果
- 同时启动后台汇总
- 汇总完成后自动刷新为新结果

#### 情况 C：没有缓存

- 进入后台汇总态
- 显示“后台汇总中”
- 显示当前正在计算的周期与日期
- 显示真实进度（已完成天数 / 总天数）
- 页面其他轻交互保持可用

### 5.4 当 `累计至今` 正在汇总时

- 如果 `近7天/近30天` 没有缓存，不允许切过去触发新计算
- 如果这些周期已有缓存，允许切过去查看旧结果
- 自定义区间不允许再打开新的计算请求

### 5.5 离开 Usage 页时

- 后台汇总继续执行
- 用户可切换到其他模块正常使用
- 再次回到 Usage 时，继续看到当前进度或最终结果

### 5.6 手动刷新

- `today` 仍支持直接刷新
- 重周期刷新也进入后台汇总，不允许在前台做重计算

## 6. 技术方案

### 6.1 首屏策略调整

- `src/App.jsx`
  - 默认模块改为 `permission`
  - Usage 在首次访问后保持挂载，支持离页后继续后台汇总

### 6.2 today 与重周期分流

#### today

- 继续保留轻量直算路径
- 仅用于今天数据
- 每分钟轻量刷新一次

#### week / month / custom

- 改为优先使用主进程 `aggregateUsageRange`
- 避免把大体积原始日志回传给渲染线程
- 聚合结果直接回到前端展示层

#### allTime

- 新增主进程 `aggregateUsagePeriod`
- 由主进程完成 `allTime` 扫描与聚合
- 前端不再直接对 `allTime` 做重扫描

### 6.3 重任务控制层

新增 Usage 内部后台任务协调层，负责：

- 单任务互斥
- 周期按钮禁用规则
- 主进程真实进度事件订阅
- 已耗时显示
- 错误兜底文案

### 6.4 缓存策略

- `today`：继续按当日与短刷新窗口判断是否过期
- `week/month/allTime`：缓存到本地，按日批次判断是否过期
- `custom`：仅保留当前页面内结果，不做长期持久化

## 7. 关键实现规则

### 7.1 不允许首屏自动触发的任务

以下任务在 Usage 首次挂载时不得自动执行：

- `week`
- `month`
- `allTime`
- `custom`

### 7.2 只允许一个重任务运行

- 重任务运行中，新重任务默认不启动
- 有缓存的周期可查看旧数据
- 无缓存的周期按钮进入禁用态

### 7.3 后台任务必须是“可解释的”

用户必须能看到：

- 正在计算哪个周期
- 当前正在处理哪一天
- 已完成多少天 / 总天数
- 已耗时
- 缓存命中天数 / 新扫描天数

### 7.4 错误恢复

- 若后台汇总失败且有旧数据，则继续显示旧数据并提示“显示上次数据”
- 若后台汇总失败且没有旧数据，则显示明确失败提示

## 8. 验收标准

### AC-01 首屏不卡死

- 新用户首次打开 App，默认不进入 Usage
- App 在普通机器上应先稳定展示轻页面

### AC-02 首次进入 Usage 只加载 today

- 首次进入 Usage 时不得同时拉起 `week/month`
- 打开 Usage 后 UI 能立即响应切换和点击

### AC-03 重周期不阻塞界面

- 点击 `近7天/近30天/累计至今/自定义区间` 时，页面不能出现整体无响应
- 用户可以在后台汇总期间切换到其他页面继续使用

### AC-04 单任务互斥生效

- 当 `allTime` 正在汇总时，无缓存的 `week/month/custom` 不应启动新任务
- 当 `week/month` 有缓存时，允许查看旧数据

### AC-05 结果自动回填

- 重任务完成后，当前页或对应缓存结果自动更新
- 不需要用户重新打开 App 才能看到结果

### AC-06 工程验证通过

- `npm run build` 通过

## 10. 版本结论

V1.4.2 Debug 的核心定位不是“把 Usage 算得更快”这一件事，而是：

- 先把首屏从重页面挪开
- 再把重周期从渲染线程挪开
- 最后把“正在算到第几天、但没死机”明确告诉用户

这是一次以稳定性和线程安全为优先级的修复版本。  
后续若继续优化，可在 `v1.4.3+` 再考虑日汇总预热、历史回填提速与更长周期缓存策略。
