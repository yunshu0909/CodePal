## 协作方式

- 你是懂技术的搭档：理解意图、主动补全技术细节、交付可运行可维护的结果。
- 沟通：解释前因后果不只甩结论；做之前先说思路；需求模糊主动追问不脑补；不确定就说不确定；每次交付说清做了 / 没做 / 还需要什么。
- 思考：先理解业务效果再定技术方案；给明确建议不和稀泥；先看能落地好维护再谈优雅；改动前评估影响范围，别改一处炸三处。

## 工作单元工作流（specs/）

每个版本 / 功能 = `specs/<版本-简述>/` 一个文件夹，全链路竖切，做完留档。`specs/` = 流动工作链，`docs/` = 沉淀知识库。

| 文件 | 阶段 | 回答什么 |
|---|---|---|
| `1-plan.md` | 计划 | 为什么做、做什么、范围、怎么拆 |
| `2-design.md` | 设计 | 设计需求总结 + 实现契约骨架 |
| `3-prd.md` | PRD | 需求规格、用户故事、验收标准 |
| `4-test-cases.md` | 测试 | 可执行测试用例 = 开发事实源 |
| → `code/` | 开发 | 读 `4-test-cases.md` 写代码 + 测试跑绿 |

- **角色边界**：`.md` = 给 AI 的事实源；`_review/` 里的 html = 给人看的视觉件，**AI 默认不读**（避免大文件占用上下文）。
- **关于 2-design**：新项目早期没有任何全局设计规范文档，这是正常的。2-design 只产出「本单元的设计需求总结 + 实现契约骨架」，**不依赖任何全局设计系统**。设计规范随迭代生长，反复稳定后才考虑沉淀成 `docs/` 下的文档——是长出来的，不是一开始就要有的。
- 规范与样板见 `specs/README.md` 和 `specs/_example-示例功能/`。

## Git 工作流

- 主干 `main`（老项目 `master` 沿用）始终保持可发布。
- 日常 trunk-based 直接提交；大 / 风险 / 实验性改动才开**短命分支** `feat/简述`、`fix/简述`，绿了 merge 回主干并删分支。
- **Conventional Commits**：`feat / fix / refactor / test / docs / chore`，scope 可选。
- 语义化发版 `vX.Y.Z`：`release: vX.Y.Z 摘要` → `git tag vX.Y.Z` → `git push && git push --tags`。**版本号只进 tag，不塞进日常 commit message。**
- 有远程才 push；纯本地项目不 push、不 tag。
- **AI 默认不擅自 commit / push / tag / reset**，除非用户明确要求。

## 文件体量信号（tripwire，非红线）

到下面量级时**停下自检：这文件 / 函数还是只干一件事吗？** 是 → 放行；不是 → 才拆。要防的是职责缠绕（改一处炸三处、bug 难定位），不是行数本身。

| 维度 | 自检信号 |
|---|---|
| 单文件 | 800 行 |
| 单函数 | 120 行 |
| 单组件状态（如 React useState） | 12 个 |

（具体数字按技术栈调整。）**真正决定拆不拆的是职责**：一个文件担 3 类以上职责 = 拆；同类逻辑第 2 次重复 = 抽公共函数。

## 本地配置安全边界

涉及本地配置 / 凭证时默认只读 + 脱敏。允许检查文件存在 / 权限 / 格式 / 字段名。**禁止输出** token / API key / cookie / secret 等真实值；示例只用假值（`sk-***`、`token_redacted`）。敏感文件（`.env` 等）不进版本库（见 `.gitignore`）。
