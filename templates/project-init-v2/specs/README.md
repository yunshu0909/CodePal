# specs/ — 工作单元

每个版本 / 功能 = 一个文件夹（工作单元），从计划到测试全链路竖着放一起。
干某个增量时只加载它那一个文件夹，全上下文齐备，不用跨目录捞。

## 结构

```
specs/<版本或功能名>/
  README.md          单元导航 + 状态
  1-plan.md          计划：目标 / 范围 / 拆解
  2-design.md        设计：设计需求总结 + 实现契约骨架
  3-prd.md           需求：用户故事 + 验收标准
  4-test-cases.md    测试：可执行测试用例
  _review/           视觉件（HTML 设计稿等），给人看
```

## 链路

`1-plan → 2-design → 3-prd → 4-test-cases → 开发`（读 `4-test-cases` 写代码 + 测试跑绿）。
前一步是后一步的输入。缺步骤就没那个文件，正常。

## 角色边界

`.md` = 给 AI 的事实源；`_review/` 的 html = 给人看，**AI 默认不读**。

## 关于 2-design（重要）

新项目刚建出来没有任何设计规范文档，这是正常的。
2-design **不依赖任何全局设计系统**，只描述「本单元的设计需求 + 实现契约」。
当某种设计模式反复出现并稳定后，才考虑沉淀成 `docs/` 下的设计文档——
那是长出来的，不是一开始就要有的。

样板见 `_example-示例功能/`（教学示例，看懂后可整个删除）。
